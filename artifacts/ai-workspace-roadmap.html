AI Workspace v0.1 → v1.0｜開發文檔

從第一版 HTML 原型升級成可用產品

本文件把每個版本要做什麼、交付什麼、不要做什麼標清楚。核心策略：不推倒重來，保留第一版 HTML 的視覺與互動概念，逐步產品化為 Web MVP、Agent Workspace，再延伸到 SwiftUI 原生端。

v0.1HTML Prototype
v0.2Web MVP
v0.3Multi-model Agent
v1.0Native + Web Stable

一、產品定位

這不是普通 ChatGPT clone，而是 AI 工作區。

Workspace→Channel→Session / Thread→Messages→Artifacts / Memory / Agent Runs

核心判斷：第一版 HTML 是「設計母版」，不是廢稿。後續版本應該把它拆成可維護元件，而不是重新想一套 UI。

二、版本路線圖

每個版本都有明確邊界，避免一開始做太大。

v0.1
已完成 / 原型階段

HTML Prototype：概念與視覺驗證

目的：確認產品長相、主畫面結構、Workspace / Channel / Session 的基本認知。

要做

  • 靜態 HTML 頁面
  • Sidebar / Chat / Artifact 三欄布局
  • 假資料展示 channel、session、message
  • 初版色彩、排版、互動感

不要做

  • 不要接資料庫
  • 不要做登入
  • 不要做複雜 agent
  • 不要在 HTML 裡硬塞完整業務邏輯

驗收標準

  • 打開頁面能看懂產品
  • 主要資訊架構清楚
  • 可作為下一版 UI 參考
交付物：index.html / 視覺原型連結
v0.2
第一個可用 MVP

Web MVP：把 HTML 產品化

目的：從靜態頁升級為可登入、可儲存、可聊天、可串模型的 Web App。

要做

  • Next.js / React / Tailwind 專案
  • Supabase Auth 登入
  • Workspace / Channel / Session CRUD
  • Messages 資料表
  • OpenAI-compatible streaming chat
  • 把 HTML 拆成 React components

不要做

  • 不要急著做 SwiftUI
  • 不要同時支援太多模型細節
  • 不要做複雜權限管理
  • 不要做插件市場

驗收標準

  • 使用者可登入
  • 可建立 workspace / channel / session
  • 訊息可保存
  • AI 可串流回覆
  • 重新整理後資料仍存在
交付物：可部署的 Web MVP
v0.3
多模型與 Agent Run

Multi-model Agent Workspace

目的：讓產品不只聊天，而是可以管理不同模型、不同 agent run、不同工作流。

要做

  • Provider adapter：OpenAI / Claude / DeepSeek / Ollama / Hermes
  • Model picker
  • AgentRun 資料表
  • run status：queued / running / done / failed
  • Tool call timeline 初版
  • Message fork 成新 session

不要做

  • 不要做完整 Dify 類 workflow builder
  • 不要讓 provider 邏輯散落 UI
  • 不要把 tool call 當純文字存

驗收標準

  • 同一 session 可切換模型
  • 每次 AI 執行都有 run 記錄
  • 失敗可追蹤原因
  • UI 能看到 agent 正在做什麼
交付物：多模型 Agent Workspace
v0.4
Artifact 與 Memory

Artifact + Memory Layer

目的：把 AI 產物與上下文沉澱下來，形成長期可用的工作區，而不是一次性聊天。

要做

  • ArtifactPanel 右側面板
  • artifact 類型：html / markdown / code / image / file
  • artifact version history
  • session summary memory
  • workspace-level memory
  • artifact 從 message / run 生成

不要做

  • 不要一開始做無限格式
  • 不要讓 memory 不可編輯
  • 不要把所有聊天都塞進 prompt

驗收標準

  • AI 可生成 artifact
  • artifact 可保存與重新打開
  • session 有摘要記憶
  • 新 session 可選擇是否引用 memory
交付物：可沉澱內容的 AI 工作區
v0.5
協作與權限

Team Collaboration

目的：從個人工具升級為團隊工作區。

要做

  • workspace members
  • role：owner / admin / member / guest
  • channel visibility
  • session sharing
  • comment / mention 初版
  • Supabase RLS 完整化

不要做

  • 不要做企業級 SSO
  • 不要做太複雜組織架構
  • 不要先做 billing

驗收標準

  • 可邀請成員
  • 不同角色權限不同
  • 私有 channel 不外洩
  • 多人可看同一 session
交付物：團隊版 MVP
v0.6
桌面級體驗

Power User UX

目的：讓產品從「能用」變成「好用、高頻使用」。

要做

  • Command palette
  • 全局搜尋
  • 快捷鍵
  • session pin / archive
  • run retry / compare
  • 模型輸出對比

不要做

  • 不要為了炫技做複雜動畫
  • 不要犧牲穩定性
  • 不要讓資訊密度過低

驗收標準

  • 重度使用者可快速切換內容
  • 能搜尋舊 session / artifact
  • 同一 prompt 可比較模型結果
交付物：高頻使用者可接受的 Web App
v1.0
穩定版

Web Stable + SwiftUI Client

目的：在 Web API 穩定後，推出 iOS / macOS / visionOS 原生端。

要做

  • 公開穩定 API
  • SwiftUI app shell
  • iOS / macOS 共用資料層
  • Session sync
  • Push notification / background run 提醒
  • Web 與原生端功能對齊

不要做

  • 不要在 SwiftUI 重寫後端邏輯
  • 不要讓原生端先於 Web 定型
  • 不要同時追求所有平台完美

驗收標準

  • Web 穩定可用
  • 原生端可登入與同步 session
  • AI run 狀態跨端一致
  • artifact 可跨端查看
交付物：正式可對外展示的 v1.0

三、建議技術架構

先 Web，後原生；先資料模型，後 agent 複雜度。

FrontendNext.js、React、Tailwind、shadcn/ui、streaming chat UI
BackendSupabase Auth、Postgres、Realtime、Storage、RLS
AI GatewayOpenAI-compatible adapter，統一 provider 格式
Agent LayerAgentRun、tool calls、status、timeline、retry
ClientsWeb first，API 穩定後再做 SwiftUI iOS / macOS / visionOS

四、核心資料表

v0.2 先建立最小 schema，後面版本再擴充。

資料表版本用途
usersv0.2使用者基本資料，接 Supabase Auth。
workspacesv0.2工作區，最高層容器。
workspace_membersv0.2 / v0.5 擴充成員與角色權限。
channelsv0.2工作區下的頻道 / 專案分類。
sessionsv0.2每個 AI 對話或任務 thread。
messagesv0.2使用者與 AI 訊息。
agent_runsv0.3每次 AI 執行、模型、狀態、輸入輸出。
tool_callsv0.3Agent 工具調用歷史。
artifactsv0.4AI 生成的 HTML、Markdown、檔案、程式碼等。
memoriesv0.4session / workspace 層級的摘要記憶。

五、第一個開發 Sprint

如果今天開始做，先做這三件事。

Sprint 1：v0.2 Web MVP 起步 1. 建立 Next.js 專案與基本 layout 2. 從第一版 HTML 拆出 components： - WorkspaceSidebar - ChannelList - SessionList - ChatThread - Composer - ArtifactPanel 3. 建立 Supabase schema：workspaces / channels / sessions / messages 4. 完成登入與 workspace 建立 5. 完成 session message 儲存 6. 串 OpenAI-compatible streaming response 7. 部署到 Vercel / Cloudflare Pages 完成標準： 使用者登入後，可以建立 workspace → channel → session，送出訊息，看到 AI 串流回覆，重新整理後資料不消失。

六、決策原則

避免走偏。

問題決策
第一版 HTML 要不要丟？不要。它是設計母版，要拆成 React components。
要不要直接 fork LobeChat / LibreChat？不建議。可以參考，但核心 Workspace / Channel / Session 應自研。
要不要先做 SwiftUI？不建議。先 Web MVP 驗證產品，再做原生端。
AI provider 要怎麼接？做自己的 AI Gateway，用 OpenAI-compatible 格式統一。
Artifact / Memory 要不要一開始做？不要。v0.4 再做，否則 v0.2 會拖太久。
