1. 產品定位

一句話：一個類 Discord / Zulip 的 AI-native workspace，用 Workspace / Channel / Session 組織 AI 對話、agent 任務、artifact、memory 與 team collaboration。

不是什麼

  • 不是單純聊天 App
  • 不是 Open WebUI clone
  • 不是 Claude Desktop clone
  • 不是 Slack bot
  • 不是純 agent runner
  • 不是普通 project management tool

是什麼

  • Discord-like communication shell
  • Zulip-like topic/session structure
  • LibreChat-like multi-model AI layer
  • AionUi-like agent session control
  • LobeChat-like AI UX polish
  • 自研 artifact / memory / task board
  • SwiftUI 原生跨 Apple 平台體驗

2. 產品核心概念

Workspace

最高層工作區，包含 members、channels、sessions、memories、artifacts、tasks、integrations、model settings。

Channel

長期分類容器，類似 Discord channel / Zulip stream。Channel 不直接等於一場對話，而是容納很多 Session。

Session

真正的 AI 工作單元：普通 AI 對話、research 任務、coding agent 任務、design 討論、artifact 生成流程。

Message

session 內訊息，類型包括 user、assistant、system、tool result、agent event、artifact event、memory event、task event。

Artifact

AI 生成或人類上傳的結果物：HTML、Markdown、Code、Diagram、Image、PDF、CSV、JSON、Prompt template。

Memory / Task

Memory 是跨 session 可引用知識；Task Board 是 AI cowork 的工作追蹤層，連結 session、agent run、artifact、message。

前端

Swift + SwiftUI，MVVM + lightweight domain layer，Observation / async-await，iOS first，macOS second，visionOS later。

後端

Supabase：Auth、Postgres、Realtime、Storage、RLS。Edge Functions 只做輕量 webhook / metadata sync。

AI Gateway

FastAPI + Pydantic + httpx + SSE。Client 不直接打模型 API，而是經過統一 adapter。

5. Supabase 資料模型

Table用途關鍵欄位
profiles應用層使用者資料，補 Supabase Authid, display_name, avatar_url
workspaces最高層工作區name, slug, owner_id, icon
workspace_members成員與角色workspace_id, user_id, role
channels長期分類容器workspace_id, name, description, kind, position
sessionsAI 工作單元workspace_id, channel_id, title, summary, status, kind, model
messagessession 內訊息role, content, content_json, status, model, parent_message_id
artifacts成果物title, kind, mime_type, storage_path, content, version
memories跨 session 可引用知識scope, kind, title, content, source_session_id, confidence, is_active
tasks任務追蹤title, description, status, priority, assignee_id, due_at
agent_runsAgent 執行紀錄session_id, task_id, status, agent_name, input, output, error

11. 決策結論

採用這條技術路線：

SwiftUI 原生前端
+ Supabase 後端
+ FastAPI AI Gateway
+ OpenAI-compatible provider layer
+ Workspace / Channel / Session 資訊架構
+ 自研 Artifact / Memory / Task Board
+ Agent run 預留，不過早複雜化

第一版最重要的不是 agent 多強，而是把核心閉環做出來：

人提出問題→AI 在 session 中工作→產生 artifact→沉澱 memory→拆出 task→task 回到 session 執行

這個閉環成立，產品就不是聊天工具，而是 AI 工作空間。
